Data migration to a NoSQL database | September - November (2024)

Context:

Project context:

The project involves migrating a relational database from a car sales company to a NoSQL architecture. This migration meets several strategic objectives, including performance improvement and addressing data loss issues. This paired project involves both technical transformation and reorganization of data modeling. The “ClassiqueModel” database includes tables related to the company's activity. The tables in the database are: Clients, Orders, Order Details, Products, Payments, Offices, and Employees.

Selection of a NoSQL model:

For this project, we chose to migrate our database to a document-oriented NoSQL model. In our situation, the document-type NoSQL model is the most suitable solution. Document databases offer great schema flexibility, making it easier to modify and add new fields. They are particularly suited for frequent updates and stand out for their scalability to handle increasing data, enabling fast and efficient processing of large volumes of information. Additionally, the document-oriented NoSQL model is less restrictive in managing complex queries and allows us to embed SQL tables into collections, avoiding complex joins.

Design of the NoSQL database model:

We decided to organize our NoSQL database into 3 distinct collections, with a Customers collection embedding two tables: Payments and Orders, as well as a sub-table Order_details contained within Orders. This was done because these tables often intersect during data analysis or database queries, minimizing joins and complex queries. Using the same logic, we created an Offices collection embedding the Employees table. We decided to keep Products in a separate collection. We considered embedding it in the Customers collection, but to preserve all products, even those never ordered, we chose to isolate Products.

Development of a pseudo-code to migrate our data from a relational database to a NoSQL database:

We developed pseudo-code to retrieve data from a relational database and build a file containing our collections and the tables within them.
