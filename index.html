Migration de données dans une base NoSQL | Septembre - Novembre (2024)

Contexte :

Contexte du projet :

Le projet consiste à migrer une base de données relationnelles d’une entreprise spécialisée dans la vente de voitures, vers une architecture NoSQL. Cette migration répond à plusieurs objectifs stratégiques, notamment l’amélioration des performances et résoudre les défaillance liées à la perte de données. Ce projet en binôme implique une transformation technique, mais également une réorganisation de la manière dont les données sont modélisées. La base “ClassiqueModel” comporte les tables liées à l’activité de l’entreprise. Les tables que l’on trouve dans la base de données sont : Clients, Commandes, Détail de commandes, Produits, Paiements, Bureaux et Employés.

Sélection d'un modèle NoSQL :

Pour ce projet, nous avons choisi de migrer notre base de données vers un modèle NoSQL de type document. En effet, dans notre situation, le modèle NoSQL de type document est la solution la plus intéressante. La base de données de type document offre une grande flexibilité de schéma, ce qui facilite les modifications et l’ajout de nouveaux champs. Elle est particulièrement adaptée aux mises à jour fréquentes et se distingue par sa capacité à évoluer facilement pour gérer l’augmentation des données, permettant un traitement rapide et efficace des volumes importants d’informations. De plus, le modèle NoSQL de type document est moins restrictif sur la gestion de requêtes complexes et nous permet d’imbriquer les tables de la base SQL en collection palliant aux problèmes de jointures complexes.

Conception du modèle de la base NoSQL :

Nous avons décidé d'organiser notre base de données NoSQL en 3 collections distinctes, avec une collection Customers où l’on a imbriqué 2 tables : Payments, Orders et une sous table Orders_details contenu dans Orders, car ce sont les tables pour lesquelles les données sont les plus susceptibles de se croiser lors d’analyses ou lectures de la base de données et d’éviter le plus possible les jointures et autres requêtes complexes. Avec cette même logique nous avons voulu former une collection Offices auquel nous avons imbriqué la table Employees. Nous avons décidé de laisser Products dans une collection à part, nous avions pensé à l’imbriquer dans la collection Customers mais pour préserver tous les produits, même ceux qui n’ont jamais été commandés nous devions isoler Products.

Nous avons réfléchit a un pseudo-code qui allait nous permettre de récupérer les données d'une base relationnelle et de contruire un fichier contenant nos collections ainsi que des tables qui se trouvent dans ces derniers

Application intermodalité | Décembre - Janvier (2024 - 2025)

Contexte :

Dans un projet en lien avec l'intermodalité pour la société SNCF, j'ai développé un logiciel avec l'aide de 6 camarades. Nous avions eu l'idée de réaliser un calculateur de trajet avec Rshiny qui avait pour but d'inciter les utilisateurs à développer un comportement plus éco-responsable, en leur proposant divers trajets possibles selon le mode de transport choisi.

Collecte de données :

Pour ce projet, nous nous sommes basé sur des jeux de données récoltés sur les site de la SNCF et RATP, telles que les positions géographiques (longitude / latitude) des stations de métros, bus, gare et parking à vélo. Nous avons également intégré des API pour la collecte de donnnées géographiques entre le point d'arrivée et de départ, mais aussi pour calculer la distance et temps (OpenRouteService) ou encore obtenir la qualité de l'air (Air Parif).

Mes contributions au sein de ce projet :

- Développement d’une carte interactive permettant la recherche d’adresses via des noms de lieux (et non des coordonnées géographiques), avec récupération automatique des coordonnées (géocodage).

- Ajout de marqueurs personnalisés avec des icônes spécifiques pour améliorer la lisibilité des points d’intérêt.

- Intégration d’une fonctionnalité "zone de couverture" : traçage d’un cercle d’un rayon défini en mètres autour d’un point, affichant tous les moyens de transport et points d’intérêt situés dans cette zone.

- Mise en place d’informations dynamiques sur les distances et les temps de trajet entre deux lieux saisis, avec prise en compte de différents modes de transport.
